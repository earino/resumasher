## [Unreleased]

### Fixed

- **OpenCode `tool_output.max_bytes` truncation detection** ([#53](https://github.com/earino/resumasher/issues/53), follow-up to the slash-command shim batch). A third end-to-end qwen3.6-35b run on OpenCode (`samples-issue42/session-ses_2359.md`) revealed the cascading defects from the earlier runs (wrong PDF filenames, missing `interview-prep.pdf`, skeletal Phase 9 telemetry) all trace to a single root cause: OpenCode caps tool output at **51,200 bytes** by default ([source](https://github.com/sst/opencode/blob/dev/packages/opencode/src/tool/truncate.ts) — `MAX_BYTES = 50 * 1024`), and resumasher's SKILL.md is now ~82KB. When the cap is too low OpenCode truncates the skill output at ~38% and replaces the rest with a meta-instruction the model usually ignores; weak local models then improvise the back half of the workflow and ship broken artifacts. The fix is a one-line opencode config change (`tool_output.max_bytes: 102400`) — confirmed in OpenCode source. `install.sh` now READS the user's opencode config (read-only — never modifies it) and prints a heads-up at install time if the cap is below SKILL.md's size, with the exact JSON snippet to add. Skipped silently when opencode isn't on PATH or the cap is already adequate. README OpenCode block adds the same prescription with the rationale (small-model truncation cascades to wrong filenames + missing PDFs + skeletal telemetry; cloud models recover gracefully so the setting is optional for them). Tests: `tests/test_opencode_command_shim.py::test_install_sh_warns_on_low_opencode_tool_output_cap` pins the install.sh detection wording, the 51200 default reference, the 102400 recommendation, and explicitly asserts the block has no write redirect into the opencode config so the read-only contract can't silently regress in future edits. Issue [#55](https://github.com/earino/resumasher/issues/55) (templated per-host SKILL.md) stays open as an optimization but is no longer required for correctness — once the cap is raised, our existing single SKILL.md fits comfortably with growth headroom.

- **Slash-command shim and weak-model prescription strengthening from the OpenCode end-to-end run** ([#53](https://github.com/earino/resumasher/issues/53), follow-up to the earlier batch). A second qwen3.6-35b run on OpenCode (`samples-issue42/session-ses_235c.md`) completed the full nine-phase pipeline end-to-end (PDFs generated, fit assessment with real strengths/gaps prose, STAR answers anchored to actual experience), but surfaced four more weak-model behaviors the SKILL.md prescriptions were too soft to prevent. **(1) `commands/resumasher.md` slash-command shim** — without a command file at `~/.config/opencode/commands/resumasher.md`, OpenCode resolves `/resumasher jd.md` by pasting SKILL.md as a user message and silently dropping the `jd.md` argument; Qwen replied "I've loaded the resumasher skill. What would you like me to do?" instead of executing. The shim is six lines (YAML frontmatter + a body that invokes the skill against `$ARGUMENTS`); `install.sh` now detects `command -v opencode` on PATH and copies the shim to the OpenCode commands directory honoring `XDG_CONFIG_HOME`. README documents the manual-copy fallback. New `tests/test_opencode_command_shim.py` pins the shim's existence, frontmatter shape, body content, and install.sh wiring. **(2) Sub-agent dispatch section reordered and strengthened** — Qwen's first task dispatch used `subagent_type="general-purpose"` (Claude Code's value, listed first in the cross-host bullet) on OpenCode and got `Unknown agent type` rejected before self-correcting to `"general"` on retry, burning ~57s. SKILL.md now leads the dispatch bullet with an explicit "use the entry that matches the CLI you're actually running in, not the first one listed" callout, names the rejection error verbatim so models recognize it, and reorders the per-host entries (Claude Code first, OpenCode second with the `NOT "general-purpose"` clarification inline). **(3) Phase 3 forbids manual fit-field writes** — Qwen bypassed `extract-fit-fields` entirely, writing each field via individual `echo "Entry/Junior" > seniority.txt` commands; that bypassed the enum validation and shipped freeform `Entry/Junior` and a 200-character recommendation prose into telemetry where neither fits any aggregation bucket. SKILL.md Phase 3 now explicitly forbids manual writes (`DO NOT replace this with `echo "8" > $RUN_DIR/fit/score.txt` etc.`) with the rationale and the run reference inline so the prescription survives future paraphrasing edits. **(4) Sub-agent prompt section forbids prompt rewriting** — Qwen rewrote the cover-letter prompt (other four sub-agents used `$PROMPT` verbatim, only cover-letter got hand-crafted), inverting "Start with a greeting H1" to "End with" and producing a PDF where the salutation rendered as a giant H1 at the bottom instead of opening the letter. The canonical template in `scripts/prompts.py` was already correct — the bug was the orchestrator not passing the compiled prompt through unchanged. SKILL.md sub-agent dispatch section now has a load-bearing "Pass `$PROMPT` AS-IS — do not paraphrase, summarize, shorten, or rewrite it before dispatching" instruction with the cover-letter regression cited as the failure mode. Filed `v0.5: Generate per-host SKILL.md from a template` ([#55](https://github.com/earino/resumasher/issues/55)) as a future architectural direction; templating would address Finding (2) by removing per-host enumeration ambiguity but doesn't touch the other three weak-model fixes here, so it's the right move for v0.5+ rather than this PR.

- **Weak-model failure modes surfaced by the OpenCode end-to-end run** ([#53](https://github.com/earino/resumasher/issues/53)). The first real `/resumasher` run under OpenCode used `qwen3.6-35b-a3b` (local Ollama-class model) — which was the whole point: weak models expose SKILL.md gaps that strong models silently paper over. The PDF resume rendered correctly but the transcript at `samples-issue42/session-ses_236d.md` showed three real shell-execution errors and one parser gap, all host-agnostic and worth fixing before merging the OpenCode port. **(1) Phase 0 `mkdir` prescription:** the SKILL.md Phase 0 example said "Write `.resumasher/config.json` with those values" but didn't show `mkdir -p .resumasher/` first. On a fresh student folder the parent directory doesn't exist, so the redirect failed with `zsh: no such file or directory: .../config.json` and the orchestrator continued with an empty config. Phase 0 now shows the explicit `mkdir -p "$STUDENT_CWD/.resumasher"` step before the `cat > ... << 'CONFIGEOF'` heredoc, with rationale in prose. New `tests/test_skill_md_weak_model_guards.py` pins the prescription. **(2) `read-resume` hard error on empty path:** when the orchestrator forgot to re-derive `$RESUME_PATH` in a fresh Bash call (variables don't persist by design across resumasher Bash tool calls), the prior call's value expanded to `""` and `"$RS" orchestration read-resume ""` arrived at the CLI. Pre-fix this silently became `Path("")` → `.` → `IsADirectoryError` deep in the call stack, while the orchestrator continued with a 0-byte `resume.txt`. Now the CLI catches empty/whitespace-only path arguments at the boundary and exits 2 with a `FAILURE: read-resume requires a non-empty path argument` message that names the likely cause (`$RESUME_PATH not re-derived`) and the recovery path. **(3) Heredoc guidance for sub-agent output writes:** the SKILL.md Phase 2 cache-the-summary example used `echo "$FOLDER_SUMMARY" > "$CACHE_PATH"` — fine if `$FOLDER_SUMMARY` was assigned via double-quoted shell, but qwen instead built `FOLDER_SUMMARY='...Ana\'s capstone...'` (single-quoted assignment with backslash-escaped apostrophes — single quotes don't allow `\'` escape inside `'...'`), so zsh died with `unmatched '` and `cache.txt` never got written. Fit-analyst then crashed with `FAILURE: ... requires variable 'folder_summary'`. Phase 2 now prescribes the heredoc form (`cat > "$CACHE_PATH" << 'HEREDOC'` — quoted delimiter is byte-literal and immune to apostrophes / dollar-signs / backticks) and explicitly forbids the broken `var='...'; echo "$var" > file` pattern by example with a "why" explanation. Same root-cause class as #44/#45/#46/#50 — SKILL.md prescribed a shell idiom that didn't survive contact with real sub-agent text. **(4) `extract-fit-fields` markdown-bold tolerance:** weak models sometimes wrap the structured-output sentinels in markdown bold (`**ROLE:** Data Analyst` instead of `ROLE: Data Analyst`) despite "on a line by itself" guidance. The pre-fix regexes only matched the plain form, so `role.txt` / `seniority.txt` ended up empty when the leading sentinel block used the bold variant — and the orchestrator then fabricated values into telemetry rather than emitting empty fields. Switched all 8 extractors (`extract_fit_score`, `extract_company`, `extract_role`, `extract_seniority`, `extract_strengths_count`, `extract_gaps_count`, `extract_recommendation`, plus `is_failure_sentinel`) to a `[\s*]*` character class around each sentinel token — matches any mix of stars and whitespace in any position, robust to all four bold variants (`**KEY:** value`, `**KEY**: value`, `KEY: **value**`, `**KEY:** **value**`) plus the plain `KEY: value` form. The lazy capture + trailing `[\s*]*$` strips trailing stars/whitespace from the value automatically. 12 new parametrized cases in `test_extractors_tolerate_markdown_bold_around_keys` covering each variant per extractor. None of these four required SKILL.md Phase 6+ changes; the test was a Tier-1 weak-model run that found Tier-1 gaps.

### Added

OpenCode (the SST one at [opencode.ai](https://opencode.ai/)) joins Claude Code, Codex CLI, and Gemini CLI as supported hosts. Concrete edits: telemetry log sniffs `OPENCODE` / `OPENCODE_VERSION` env vars and `~/.opencode/skills/` for install-scope detection (host string `opencode_cli` lands in dashboards correctly); SKILL.md path resolution prologue searches `~/.opencode/skills/resumasher` and project variants; SKILL.md cross-host enumerations at lines 130-138 (question tool), 237-241 (sub-agent dispatch), 460 (web fetch), 666 (web search), 708 / 734 (parallel sub-agent dispatch with note that OpenCode parallel was historically flaky per [sst/opencode#14195](https://github.com/sst/opencode/issues/14195) and may serialize), 790 (question batching) all add the OpenCode bullet; `DEFAULT_IGNORE_DIRS` in `scripts/orchestration.py` adds `.opencode` so a student with a project-scope OpenCode skill install doesn't get the skill source mined as evidence; README adds an OpenCode install block (defaulting to the Claude-compat path for simplicity, with native `~/.opencode/skills/` install documented as the alternative) and an OpenCode update block alongside the existing three; PRIVACY.md updates the install-path enumerations and host list; tests add `OPENCODE_VERSION` env-var sniffing assertion in `test_telemetry_scripts.py`, install-scope auto-detection assertion for `~/.opencode/skills/`, and `.opencode` to the parametrized self-mining-risk regression in `test_orchestration.py`. Decisions explicitly skipped to keep scope tight: install.sh stays document-only for OpenCode (Claude-compat path is the recommended default, no install.sh branching needed); inline-prompt sub-agent dispatch stays the model (the Codex pattern), no `.opencode/agents/<name>.md` files shipped — OpenCode's built-in `general` subagent_type accepts arbitrary prompts at runtime, identical to how Codex handles dispatch. Requires OpenCode v1.0.110+ for native skill discovery.

## [0.4.3] — 2026-04-26

v0.4.3 closes a class of bugs surfaced by real-run testing of v0.4.2: SKILL.md prescribed (or allowed) shell patterns that didn't survive contact with real data. JSON-on-stdout corrupted by `echo` backslash interpretation ([#44](https://github.com/earino/resumasher/issues/44)), sub-agent prompts leaking PII into world-readable `/tmp` ([#45](https://github.com/earino/resumasher/issues/45)), tailor LLM occasionally combining two projects under one H3 heading ([#46](https://github.com/earino/resumasher/issues/46)), and `fit-extracted.env` shell-source dropping field values that contain spaces ([#50](https://github.com/earino/resumasher/issues/50)). All four fixed structurally — the SKILL.md idioms now route through narrow CLI commands or per-field files instead of round-tripping through shell strings, so the bug class is eliminated for these endpoints, not just the specific occurrences. Plus the typography overhaul from [#42](https://github.com/earino/resumasher/issues/42) — stacked dates, real italics via bundled DejaVu Sans Oblique, body-wide URL linkification, structural section dividers, per-sub-block `KeepTogether` for correct page breaks, and project-title collapse for `Name (URL)` shapes. Closes the last open piece of [#22](https://github.com/earino/resumasher/issues/22) (right-aligned dates, reframed as stacked-line for ATS safety) and [#31](https://github.com/earino/resumasher/issues/31) (LaTeX renderer track) as not-planned. No breaking changes; students on v0.4.2 upgrade in-place.

### Fixed

- **`parse-job-source` JSON-on-stdout replaced with two narrow CLI commands** ([#44](https://github.com/earino/resumasher/issues/44)). The original Phase 1 command emitted `{"mode":..., "path":..., "content":...}` as JSON on stdout. When a shell that interprets `echo` backslash escapes (zsh, dash, bash with `xpg_echo` set, busybox) consumed that output via `PARSED=$(...); echo "$PARSED" | python3 ...`, the JSON's `\n` got rewritten to a real newline before parsing — producing a `JSONDecodeError: Invalid control character` traceback on every macOS resumasher run. Replaced by `parse-job-mode` (single word stdout: `file`/`url`/`literal`) and `parse-job-content` (raw bytes stdout: file text / URL / literal text). The new SKILL.md Phase 1 idiom captures mode in a shell variable and pipes content directly through `format-jd` — no JSON ever round-trips through a shell string, so the bug class is structurally eliminated for this endpoint. `parse-job-source` removed entirely (no external callers; SKILL.md is the only consumer). New tests: per-mode CLI output assertions plus a load-bearing bash-pipeline regression test that runs the new SKILL.md idiom in a real `bash -c` subprocess with a JD containing curly quotes, em-dashes, and ligatures (the actual real-run shape that surfaced the original bug), asserting byte-perfect round-trip from input file → format-jd → output file.
- **Sub-agent prompts no longer leak student PII into `/tmp`** ([#45](https://github.com/earino/resumasher/issues/45)). Real-run testing of #43 surfaced that some agents (Claude Code's Bash tool was the observed offender on macOS) improvised around SKILL.md's prompt-build pattern: instead of capturing the rendered prompt in a shell variable (which can hit argv-length caps for the folder-miner prompt at 100KB+), they wrote it to `/tmp/<kind>-prompt.txt` and passed the path. That leaked the student's resume + JD + project content as plaintext PII into a directory that's world-readable on macOS (mode 0755) until reboot. Belt-and-suspenders fix: **(suspenders)** SKILL.md's "Sub-agent prompt pattern" section now explicitly prescribes `$RUN_DIR/prompts/<kind>.txt` for prompt staging — gitignored, run-scoped, wiped each run — and explicitly forbids `/tmp/` for this purpose with the privacy reasoning called out so future maintainers don't drop it. **(belt)** New `cleanup_stray_prompts()` function + `cleanup-stray-prompts` CLI subcommand modeled on `cleanup_stray_outputs` from #29. Scans `/tmp` (top-level only, never recursive) for files whose basenames match `<kind>-prompt.{txt,md}` or `<kind>_prompt.{txt,md}` for a registered prompt kind AND whose mtime is newer than `$START_TS`, then deletes them unconditionally (no recovery value — they're transient sub-agent intermediates containing PII we want gone). Wired into SKILL.md Phase 8 alongside the existing `cleanup-stray-outputs` invocation. Pattern matcher accepts both hyphen-separator (`folder-miner-prompt.txt`) and underscore-separator (`folder_miner_prompt.txt`) stem forms — both shapes appeared across different real-run sessions. 16 new deterministic tests covering each registered kind across both separators, mtime gate, suffix gate, top-level-only scope, generic /tmp file immunity (bash_history, screenshots, near-miss names), missing scan dir, CLI JSON summary, and SKILL.md prescription assertions (the prescribed `$RUN_DIR/prompts/` line + explicit forbid of `/tmp/` + the cleanup invocation must all stay in SKILL.md across future edits).
- **Tailor must emit one project per H3 heading** ([#46](https://github.com/earino/resumasher/issues/46)). Real-run testing of #43 surfaced a tailor LLM occasionally combining two thematically-related GitHub repos under a single H3 — `### prompt-harness + nonprofit.ai (github.com/me/a, github.com/me/b)` — instead of emitting separate `### Name (URL)` blocks. Combined headings have two failure modes: ATS parsers expect one project per heading and the long combined string confuses extraction; the renderer's title-collapse logic only handles `Name (single URL)` and falls back to inline-URL rendering on multi-URL parens, so combined headings render visually inconsistent with the canonical project entries in the same resume. Fix is a tailor prompt change in `scripts/prompts.py` — the Projects section now contains an explicit "One project per H3 heading" rule with concrete combiner examples to avoid (`+`, `&`, `/`) plus the affirmative "emit two separate `###` blocks instead" instruction. Schema-block annotation `<-- ONE project per heading` placed inline with the H3 line for maximum salience to the LLM. 4 new deterministic tests pin the rule + combiner examples + affirmative behavior + schema annotation in the template against future drops. New Tier-3 live test (`tests/test_tailor_combined_projects_live.py`) invokes `claude -p` against Opus 4.7 with a synthetic candidate whose folder summary describes two thematically-related repos in maximum tailor-tempting detail (one CLI tool + one static site that consumes its output) and asserts Opus emits separate H3 blocks. Verified in sandbox before merge — Opus produces two separate `### Name (URL)` headings, no combined heading. Auto-skips when `claude` is absent (CI runners) or `RESUMASHER_SKIP_LIVE=1`. Synthetic Sam Jones fixture (resume.md + jd.txt + cache.txt) under `tests/fixtures/tailor-combined-projects/` — fully fabricated, safe to commit.
- **Fit values with spaces no longer get silently dropped via shell-source corruption** ([#50](https://github.com/earino/resumasher/issues/50)). Real-run testing of #43/#47/#48/#49 surfaced agents improvising an `fit-extracted.env` heredoc + `source` pattern between Phase 3 (fit extraction) and Phase 9 (run_completed telemetry). When the company name contained a space (`COMPANY=Elevation Capital`, unquoted on its own line), bash parsed `Capital` as a command name, the assignment was discarded, and `$COMPANY` ended up empty — `run_completed` shipped with `--company ""` and `history.jsonl` recorded `"company": ""`. Same root-cause class as #44/#45/#46: SKILL.md prescribed (or allowed) a shell pattern that didn't survive contact with real data. Fix: new `extract-fit-fields --output-dir <dir>` CLI command that reads fit-assessment text on stdin and writes one file per field under `$RUN_DIR/fit/` (`score.txt`, `company.txt`, `role.txt`, `seniority.txt`, `strengths.txt`, `gaps.txt`, `recommendation.txt`). Phase 9 reads each via `$(cat $RUN_DIR/fit/<field>.txt)` — strips the trailing newline but preserves all interior characters byte-perfect (spaces, ampersands, single quotes, dollar signs, backticks, em-dashes). SKILL.md updated end-to-end: Phase 3 now invokes `extract-fit-fields` and persists `$OUT_DIR` to `$RUN_DIR/out-dir.txt` for Phase 9 reuse; Phase 9's run_completed and append-history code blocks now show the `cat`-pattern explicitly with a "do NOT improvise an env-file source step" callout. 8 new deterministic tests including a real-bash regression that asserts `Elevation Capital`, `Head of AI & Product`, and an apostrophe-and-dollar-sign edge case (`Macy's & $hop`) all survive write+read round-trip byte-perfect; plus SKILL.md text assertions that the prescription stays in place and the prohibited shell-source patterns never reappear.

### Changed

- **Stacked-date layout for resume PDF block titles** ([#42](https://github.com/earino/resumasher/issues/42), closes [#22](https://github.com/earino/resumasher/issues/22), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max) — #22 surfaced six concrete typography complaints, four shipped in v0.4.0 via #28; this is the last one). Block and sub-block titles that contain a recognizable date segment now render as two paragraphs — title above, date below in a slightly smaller (9.5pt) regular-weight metadata style — instead of inlining the date in parentheses. Targeted fix for the last remaining piece of #22 (right-aligned dates), reframed for ATS safety: single-flow text only, no Tables or tab stops, because [Jobscan testing](https://www.jobscan.co/blog/resume-tables-columns-ats/) finds 2-column / Table layouts are the most common cause of word-salad parsing in commercial ATS systems (Workday, Greenhouse, Taleo, iCIMS). Resumasher's north-star metric is "the application gets through the parser," not "the PDF could win a typography award" — so we trade the literal right-aligned ask for a stacked-line variant that preserves the typographic gesture (real size hierarchy between title and metadata) while reading linearly to every parser. Detection heuristic: parenthesized-at-end or pipe-separated segment containing a 4-digit year (1900-2099). Bare years in prose ("2024 Economic Survey — Statistics Austria") are NOT matched, so publication titles aren't hijacked. Titles without a recognizable date fall through to single-paragraph rendering — zero behavior change in that case. New `BlockMetadata` and `SubBlockMetadata` paragraph styles (9.5pt regular, body-black, sub-block indent matches `SubBlockTitle`). **Bundled correctness fixes:** the stacked-date pattern surfaced two pre-existing layout problems that would have shipped a regression on multi-role resumes — bullets under sub-blocks rendered with the bullet character to the LEFT of the sub-block content above (Bullet.bulletIndent=4 vs SubBlockTitle.leftIndent=8, glaring once a metadata line sat directly above the bullet), and SectionHeading.spaceBefore=10 felt proportionally compressed once each block gained a metadata line. Both fixed alongside: new `SubBlockBullet` paragraph style with leftIndent=22 / bulletIndent=12 used for bullets under sub-blocks, and SectionHeading.spaceBefore bumped 10→14 to restore visible separation between sections. Top-level Bullet style unchanged (regression-guarded) so resumes without sub-blocks see no layout shift. **Structural section dividers:** each section heading (Summary / Experience / Education / Skills / Projects / etc.) now has a thin horizontal rule (`HRFlowable`, thickness 0.75pt, color `#333333`) directly underneath it, so sections read as visually-separated structural blocks rather than a wall of stacked text. Distinct from the in-content `---` markdown rule (`thickness 0.5pt`, color `#888888`) which is a softer in-section break — section dividers are slightly thicker and darker to read as structural. ATS round-trip preserved: `HRFlowable` produces no text in the PDF text layer, so parser extraction is unaffected. Also applied to interview-prep document rendering where it uses `SectionHeading`. **Polish pass:** `BlockMetadata` and `SubBlockMetadata` colored `#555555` so date/location lines recede as metadata rather than competing with bullets for attention; `Contact` and `SubheadCenter` (US-style centered contact) reduced to 9.5pt and recolored `#555555` so the contact line reads as header rather than body; `BlockTitle.spaceBefore` bumped 4 → 6 for natural breathing room between consecutive jobs within a section. **Clickable links anywhere a URL appears:** `_linkify_contact()` was generalized to `_linkify_text()` and applied to every Paragraph render path — block titles (so a project entry like `### Resumasher (github.com/earino/resumasher)` produces a clickable link), sub-block titles, bullets (so a bullet like "see https://example.com/docs" is clickable), and Summary-style body paragraphs. Pattern set unchanged: email, `linkedin.com/in/X`, `github.com/X`, explicit `https://...` URLs. Bare-domain forms get auto-prefixed with `https://` on the href while displayed text stays bare-domain. Phone, location, prose, and any non-URL text pass through unchanged. **Page-break correctness via per-sub-block `KeepTogether`:** real-run testing surfaced that a multi-role block (e.g., one company with 3 sub-roles, each with its own bullets) was wrapped in a single giant `KeepTogether` flowable. When the entire block exceeded the remaining page space, reportlab page-broke BEFORE the whole thing — leaving the bottom of page 1 blank. Fixed by emitting per-sub-block `KeepTogether` groups: block title (+ direct bullets, if any) gets one group, each sub-block (+ its bullets) gets its own. Page breaks now occur between sub-blocks — the natural typesetting boundary — while sub-role titles stay glued to their own bullets. Single-role blocks (no sub-blocks) still emit exactly one `KeepTogether` each — no over-fragmentation. **Final visual polish:** (1) markdown code-span backticks (the tailor LLM commonly wraps URLs in `` ` ``-pairs like `` `github.com/foo` ``) are now stripped at render time so the rendered output has no decorative backticks; reportlab Paragraph doesn't render code spans natively and there's no use for the markers in a resume. (2) Link `<a>` tags now carry an explicit `color="#0645AD"` (muted Wikipedia-blue) so URLs are visually discoverable to recruiters skimming the PDF rather than blending with body text — earlier "minimalist, links inherit color" call was reversed after real-run review showed clickable URLs went unnoticed. (3) URL regex character class tightened to exclude every character RFC 3986 says is never in a URL (whitespace, `<>"`{}^|\`) plus closing paren / bracket — backtick exclusion specifically fixed a bug where href values gained a trailing `` ` `` baked into the link target. **Project titles shaped `Name (URL)` collapse to a single clickable Name.** Pre-fix, a project entry like `### ci_fixer_bot (github.com/earino/ci_fixer_bot)` rendered as `ci_fixer_bot (github.com/earino/ci_fixer_bot)` with only the URL portion in blue — explicit URL display next to the project name was visual noise. Now the entire title becomes a single clickable Name in muted blue (the URL still lives in the `<a>` href attribute, ATS extraction still finds the link). New `_linkify_title()` (used only in titles, not bullets — bullets routinely have parens with extra metadata like star counts and we want to keep the inline-URL render there). Title-collapse triggers only when parens content is EXACTLY one of our URL patterns; titles like `Project (Aug 2022 – Aug 2025)` (date in parens, handled by date split) and `Project (github.com/me/x, 23 stars)` (URL plus extra prose) are unaffected. **Date split now works for titles with trailing text after the parens.** Pre-fix, the parens-with-date regex was anchored to end-of-string (`\s*$`), so a title shaped `Senior Director, Data Science (August 2022 – August 2025) — Zurich` (closing paren NOT at end — `— Zurich` trails it) silently failed to split: the date stayed inlined and only titles with parens at the very end (CEU-style: `Visiting Professor (December 2017 – July 2025)`) got the stacked-date treatment. The end-anchor was wrong from the start; real-run resumes routinely have a city/location between sub-blocks and the parent block. Removed the anchor and changed the title rebuild to preserve text on BOTH sides of the parens — title becomes `Senior Director, Data Science — Zurich` and the metadata line carries `August 2022 – August 2025`. Five regression tests pin the shape including parens-at-start, parens-at-middle, parens-at-end, and an end-to-end render of the exact broken case from Eduardo's run. **Italic markdown finally renders as italic.** Pre-#42 the renderer skipped `*italic*` conversion for fear of false-matching single asterisks in prose ("footnote*"), so course titles like `*Applied Deep Learning*` (common in tailored bullets) shipped with the asterisks visible as decorative characters. Now: `*text*` → `<i>text</i>` via a regex that requires non-whitespace immediately inside the `*` pair (eliminates math-operator false positives like `5 * 4`) and forbids the pair from spanning a newline. AND we bundle `DejaVuSans-Oblique.ttf` + `DejaVuSans-BoldOblique.ttf` (~1.3MB total) in `assets/` and wire them via `registerFontFamily(italic=...)` so `<i>` tags actually render as visually-italic, not just markers-stripped regular text. Earlier "save 1.4MB on repo size" call (PR #30, deferred to a future LaTeX track) was the wrong tradeoff — visible asterisks in shipped resumes vs. negligible repo growth alongside a 50MB venv. Link styling: no `color=` attribute on the `<a>` tag, so links inherit the surrounding paragraph color (the soft gray of the contact line) — no 1990s blue underline, modern resume convention. Verified end-to-end: PDF Link annotations actually present in the output PDF (pdfminer pulls `mailto:`, `https://linkedin.com/...`, `https://github.com/...` from the rendered file). ATS round-trip preserved — link annotations are a metadata layer; the underlying text stream is identical to the pre-link version. 14 new deterministic tests including pattern recognition for parens/pipe/year-range/Present forms, no-date regression guard, sub-block treatment with correct indent, ATS round-trip assertion that all date strings remain extractable in document order, sub-block bullet indent invariant, top-level bullet regression guard, and section-separation lower bound. Closed [#31](https://github.com/earino/resumasher/issues/31) (LaTeX renderer track) as not-planned in the same arc — engine swap was disproportionate to the actual problem.

### Fixed

- **interview-coach sub-agent no longer pollutes student CWD on weaker models** ([#29](https://github.com/earino/resumasher/issues/29)). On Haiku 4.5 (does not reproduce on Opus 4.7), the interview-coach sub-agent ignored the buried `TOOL USAGE CONSTRAINTS` block and used the Write tool to plant a markdown file with a fabricated name (e.g. `Ana_Muller_Interview_Prep_Bundle.md`) directly in `$STUDENT_CWD` — a hard contract violation, since resumasher's documented output location is `applications/<slug>-<date>/`. Defense-in-depth fix across four surfaces: **(1) Prompt surgery** in `scripts/prompts.py::INTERVIEW_COACH_PROMPT` — opening changed from "Build an interview preparation bundle" to "Generate interview questions and answers", added explicit "Return the document as text in your response. Do not create any files on disk." to the first paragraph, moved the `TOOL USAGE CONSTRAINTS` block from the bottom (line ~550 of a 10KB prompt, AFTER the "produce a markdown document" framing had already shaped the model's plan) to immediately after the first paragraph, and strengthened the constraints to specifically name the Write tool. **(2) SKILL.md Phase 6 hardening** — replaced ambiguous "Save the outputs to `$OUT_DIR/cover-letter.md` and `$OUT_DIR/interview-prep.md`" with explicit instructions: take the sub-agent's text response and use the Write tool to save it; do NOT scan the filesystem for sub-agent-written files (that behavior IS the bug, not the recovery); reference issue #29 so future edits don't revert the explicit wording. **(3) Post-phase cleanup scan** — new `orchestration cleanup-stray-outputs` subcommand and `cleanup_stray_outputs()` Python function. Scans `$STUDENT_CWD` top level (never recursive) for `.md` files whose names contain "interview", "prep", or "bundle" (case-insensitive substring match) AND whose mtime is newer than the dispatch timestamp (passed via `--since-timestamp`). Pre-existing student files are immune via the mtime gate; documented input filenames (`resume.md`, `cv.md`, `jd.md`, etc.) are immune via a protected-names allowlist. If `$OUT_DIR/interview-prep.md` is missing/empty, MOVE the rogue file there (best-effort content recovery); otherwise DELETE it (orchestrator already wrote the right thing; rogue is pollution). Wired into SKILL.md Phase 6 to run after every interview-coach dispatch. **(4) Test coverage** — three new test files: `test_cleanup_stray_outputs.py` (12 deterministic tests covering DELETE/MOVE paths, mtime gate, name-pattern matching, protected-names guard, top-level-only scope, defensive missing-input handling, and CLI subcommand integration), `test_interview_coach_prompt.py` (6 structural assertions on the prompt text — first-paragraph framing, no-files imperative, constraints-before-emit ordering, Write-tool-named-explicitly, substitution-slot preservation, FAILURE-sentinel preservation), `test_skill_md_phase6.py` (5 wording assertions on SKILL.md). Plus a Tier 3 live LLM test (`test_interview_coach_live.py`) that invokes `claude -p --model claude-haiku-4-5` against the real interview-coach prompt with a synthetic-persona fixture set (Ana Müller, MS Business Analytics at CEU Vienna — fully fabricated, safe to commit) and asserts zero Write tool calls + zero stray markdown files. The live test auto-skips when `claude` is absent (CI runners) or `RESUMASHER_SKIP_LIVE=1` (manual escape hatch); registered via the `live_llm` pytest marker in `conftest.py`. Runs automatically on dev boxes that have Claude Code installed; uses the developer's existing subscription so per-run cost is zero. The cleanup scan is the belt (catches rogue files even if a future model regresses against the prompt); the prompt surgery is the suspenders (stops Haiku from creating the rogue file in the first place); the live test is the one-time signal that the suspenders work. Per-run human verification on Haiku before merge is the acceptance criterion the issue asks for.
- **`discover_resume` preserves on-disk filename case** ([#27](https://github.com/earino/resumasher/issues/27)). On case-insensitive filesystems (macOS APFS default, Windows NTFS default — so ~100% of the cohort), `scripts/orchestration.py::discover_resume` iterated `RESUME_CANDIDATES` and returned `cwd / "cv.pdf"` when the on-disk file was actually `CV.pdf`: the probing `exists()` check matched case-insensitively but the returned `Path.name` reflected the candidate string we constructed, not the real filename. Functionally it still resolved to the right file (reads worked), but logs and telemetry showed the wrong case. Fix: enumerate the directory with `iterdir()` once, match on lowercased names against a priority map built from `RESUME_CANDIDATES`, and return the highest-priority match with its real on-disk `.name` intact. Priority order (markdown > PDF, resume > CV) is preserved. New regression test `test_discover_resume_preserves_on_disk_case_mixed` uses a name (`Cv.pdf`) that isn't literally in `RESUME_CANDIDATES`, so it catches the bug on case-sensitive filesystems (Linux CI) too — the old probing loop would have returned None. The existing `test_discover_resume_accepts_cv_pdf_variants` tightened from a case-insensitive `.lower() == "cv.pdf"` check to a strict `== "CV.pdf"`, which fails on the old code under Windows NTFS — the Windows CI matrix merged in #35 is now the regression gate where it matters most.
- **Windows test harness — telemetry tests invoke scripts via explicit `bash`.** Surfaced by the Windows CI matrix ([#34](https://github.com/earino/resumasher/issues/34)): `tests/test_telemetry_scripts.py::_run` invoked `bin/resumasher-telemetry-log` (a bash script) directly via `subprocess.run([str(bin_path), ...])`. On POSIX the kernel honors the `#!/usr/bin/env bash` shebang, but Python's `CreateProcessW` on Windows can only launch PE/EXE files — the script invocation failed with `OSError: [WinError 193] %1 is not a valid Win32 application` and ~25 tests errored out. Git Bash has its own shebang interpreter, but `subprocess.run` doesn't go through Git Bash. Fix: prepend `"bash"` to the argv so Python launches `bash.exe` (Windows) or `/bin/bash` (POSIX) and bash itself handles the script. Matches how students actually invoke the scripts. POSIX test behavior is unchanged. Production path was unaffected — `bin/resumasher-exec telemetry` dispatches through bash-to-bash, no Python subprocess involved.
- **Windows test harness — install-probe regression test skipped on Windows.** `tests/test_install_python_probe.py::test_install_sh_falls_through_when_python3_is_broken_ms_store_stub` simulates a Windows-only bug (Microsoft Store `python3` stub) on non-Windows by dropping fake bash-script shims on PATH. Python's `subprocess.run` on Windows can't execute bash scripts (same WinError 193 as above), so the test has no way to run its own simulation natively. Marked `@pytest.mark.skipif(sys.platform == "win32")`. Windows students still catch regressions against real MS Store stubs organically; this test provides Linux/macOS-side regression coverage.
- **Windows stdout/stderr encoding — CLIs no longer crash on `→`, `…`, or non-ASCII names.** Surfaced by the new Windows CI matrix ([#34](https://github.com/earino/resumasher/issues/34)): `scripts/orchestration.py build-prompt` (and the other CLI entry points) wrote prompt templates containing `→` directly to stdout, but Python on Windows defaults stdout/stderr to the system ANSI code page (typically CP1252) when not attached to a TTY — which is exactly the shape every `resumasher-exec` invocation takes. Writing `→` raised `UnicodeEncodeError: 'charmap' codec can't encode character '→'` and the phase died. Any Windows student running orchestration whose output included a `→`, `…`, curly quotes, or a non-ASCII name hit this — the v0.4.1 install fixes got students past install but not past the first CLI call that produced these glyphs. Fix: call `sys.stdout.reconfigure(encoding="utf-8")` (and the same for stderr) at every `if __name__ == "__main__":` block in `scripts/` — orchestration.py, render_pdf.py, github_mine.py. New `tests/test_stdout_encoding.py` reproduces the Windows behavior on any OS via `PYTHONIOENCODING=cp1252`, pinning both the crash-free invariant AND the byte-level assertion that the arrow glyph makes it through as UTF-8 (not lossily transcoded to `?`).

## [0.4.1] — 2026-04-24

v0.4.1 is a Windows hotfix release. The cohort is ~90% Windows, and [@b0glarka](https://github.com/b0glarka) reported [#32](https://github.com/earino/resumasher/issues/32) — a fresh Windows 11 + Git Bash install of v0.4.0 failed before the skill could run. Three cascading Windows-specific bugs in `install.sh` were caught across two rounds of verification on her machine; all three are fixed here. Windows students on a fresh clone can now run `bash install.sh` end-to-end.

### Fixed

- **Windows (Git Bash) install and runtime** ([#32](https://github.com/earino/resumasher/issues/32), reported by [@b0glarka](https://github.com/b0glarka) with a three-file patch that pinpointed every failure site, then caught two additional Windows-specific failure modes during verification of the patch itself). Three cascading Windows bugs fixed together. **(1) Venv layout and interpreter name:** `install.sh` hardcoded `python3` and `.venv/bin/pip`; `bin/resumasher-exec` hardcoded `.venv/bin/python`; both SKILL.md prologues gated on `.venv/bin/python`. On Windows/Git Bash, `python3` is typically absent and CPython's venv layout uses `Scripts/` instead of `bin/`. Installer now prefers `python3` and falls back to `python`; `VENV_BIN` is derived as `bin/` or `Scripts/` after venv creation; wrapper and both SKILL.md prologues OR-join the POSIX and Windows venv-Python checks. **(2) Microsoft Store App Execution Alias stub:** Windows also ships a `python3` stub at `%LOCALAPPDATA%\Microsoft\WindowsApps\python3` — a real file on PATH that exits non-zero with "Python was not found" when invoked. A pure `command -v` check accepts the stub and aborts at `python3 -m venv`. Installer now probes each candidate by running a 3.10+ version check, disqualifying the stub and falling through to `python`. **(3) pip self-upgrade via the pip binary:** On Windows, `pip.exe` can't overwrite itself while running (file is locked), and pip detects this and refuses `pip install --upgrade pip` when invoked via the binary — redirecting to `python -m pip install --upgrade pip`, which copies pip to a temp path before replacing. Installer now invokes pip via `python -m pip install` uniformly, sidestepping the self-upgrade wall. `tests/test_install_python_probe.py` simulates the MS Store stub on any OS and pins both the probe-by-invocation and `python -m pip` invariants. No Windows CI matrix yet — the reporter smoke-tested the patch on her Windows 11 + Git Bash setup and surfaced both the MS Store stub bug and the pip-self-upgrade bug before merge.
- **SKILL.md prologue drift guard.** The two path-prologue blocks (top-of-file and the re-render section at L1026) had already silently drifted before #32 — Copy B was missing the "venv missing → run install.sh" error branch that Copy A had. Any cross-cutting fix (like the Windows one above) has to touch both, and "forgot the second copy" is a silent-break class of bug. New `tests/test_skill_md_prologue.py` asserts exactly two prologues exist, both check POSIX and Windows venv-Python paths, and both scan the same candidate install roots. Future cross-cutting changes that touch only one copy will fail in CI instead of shipping.

### Fixed

- **Photo area: aspect preserved, placement configurable, horizontal rules rendered** ([#22](https://github.com/earino/resumasher/issues/22), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Three visible PDF fixes shipped together. (1) **Aspect ratio:** the renderer previously hard-coded `3cm × 3cm`, stretching portrait photos by up to 33% (students described it as "flattened" or "like a funeral-hall portrait"). Now computes width/height from source dimensions, clamping the longer side to 3cm and preserving aspect. (2) **Placement:** photos were always center-aligned at the top (reportlab's `Image.hAlign` defaults to `CENTER`). New `photo_position` config field, `--photo-position` CLI flag, and first-run setup question (right / left / center, default right = DACH convention). Existing users with `include_photo: true` get asked once on next run. (3) **Horizontal rules:** markdown `---`, `___`, or `***` on its own line between section content now renders as an actual hairline rule (`HRFlowable`) instead of being silently dropped. Full-line match only so inline `---` in prose stays as text. 14 new tests cover each invariant. Typographic polish (the taste-call half of #22 — font hierarchy, right-aligned dates) was attempted in PR #30 and closed unmerged: reportlab-by-hand typography re-derived with padding and color what LaTeX templates already solve with real font families, proper hinting, hanging indents, and hyphenation. Deferred to [#31](https://github.com/earino/resumasher/issues/31) (optional LaTeX renderer as a second engine).
- **Photo path persisted to tailored markdown** ([#20](https://github.com/earino/resumasher/issues/20), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Previously, a student who provided a photo got it embedded in `resume.pdf` but the source `tailored-resume.md` had no record of the photo path — re-render after manual markdown edits required config or a CLI flag, making the markdown non-self-describing. The tailor now emits `<!-- photo: /path/to/photo.jpg -->` in the markdown header when a photo is provided; `parse_resume_markdown` extracts the path onto `ResumeDoc.photo_path`; the renderer uses it when `--photo` isn't explicitly passed. HTML comments are invisible in markdown previews (no visual noise) but captured reliably by the parser. Precedence: explicit `--photo` flag > markdown comment > no photo. Twelve tests guard the precedence rules plus comment parsing (whitespace tolerance, duplicate-comment-first-wins, inline-bold safety, `format_contact_info` emission).
- **Local test suite now works in a macOS venv** ([#24](https://github.com/earino/resumasher/issues/24)). Tests that shell out via `subprocess.run(["python", "-m", ...])` were spawning a subprocess that resolved `python` through `$PATH`, which typically finds system Python (not the venv's). The venv's installed packages (reportlab, pdfminer.six, etc.) weren't visible to the subprocess, and ~15 tests failed with `ModuleNotFoundError`. CI didn't catch this because GitHub Actions puts the matrix Python on `$PATH`. Fix: replace every `"python"` with `sys.executable` in subprocess calls, so the subprocess always uses the same interpreter as the test runner. No behavior change in CI; local test runs now just work.
- **Bullets stay attached to their project titles** ([#19](https://github.com/earino/resumasher/issues/19), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Previously, when the tailor emitted `**Project Name** | metadata` directly under `## Research Experience` (or Work Experience, or similar) without an intervening `### Company` heading, `parse_resume_markdown` silently dropped the titles into `raw_paragraphs` and bullets into `raw_bullets`. The renderer stacked all project titles at the top of the section then dumped all bullets at the bottom in a single flat list — readers could not tell which bullet belonged to which project. The parser now synthesizes a `ResumeBlock` when it sees `**Title**` with trailing metadata directly under `##`, and bullets attach to that synthetic block until the next `**Title**` or `##`. The trailing-metadata check guards against hijacking bold-only prose in Summary sections (e.g., `**Accomplished data leader.**` still renders as a paragraph, not a block title). Existing `### Company` + `**Role**` sub-block behavior for multi-role tenures is unchanged. Six new parser + renderer tests guard against regression, plus the previously-fired `ORPHANED_BULLETS` inspect warning is now proven to stop firing for this shape.
- **Rendered PDFs can no longer ship without a candidate name** ([#18](https://github.com/earino/resumasher/issues/18), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Previously, when the tailor emitted a pipe-separated contact line on line 1 without a `# Name` H1 prefix, `parse_resume_markdown` silently dropped the line, producing a PDF with no name, email, phone, or LinkedIn anywhere — an ATS cannot identify the applicant, the application is silently filtered out, the student never hears back. The renderer now raises `MissingContactHeaderError` before writing any PDF, with a student-facing error that names the expected shape, shows the first line the parser actually saw, and points at KNOWN_FAILURE_MODES.md #1 for the tracking issue. Tailor prompt hardened to explicitly forbid any other line-1 shape. Exit code 2 on failure so orchestration knows to surface the error. Five new regression tests in `tests/test_render_pdf.py` guard against reintroducing the silent-drop behavior.

- **`pg_cron` retention job.** Events older than 90 days and installations with no activity for 180 days are deleted daily at 03:00 UTC. Runs inside Supabase — no external scheduler needed. Aggregate dashboard views survive retention.
- **`supabase/` source of truth.** Applied migrations + both edge functions + public config committed to the repo so the backend state is auditable.
- **Host self-reporting (`--host`)** alongside model self-reporting. Codex CLI doesn't set env vars that bash can sniff, so the orchestrator passes `--host codex_cli` literally. Same pattern as `--model`. Resolution order: flag > RESUMASHER_HOST env > env-var sniff > "unknown". Result: 100% `host` field population across Claude Code / Codex / Gemini in live tests.
- **Scope-matched state directory.** User-scope install (skill at `~/.claude/skills/...`) → state in `~/.resumasher/`. Project-scope install (skill at `<project>/.claude/skills/...`) → state in `<project>/.resumasher/`. Deleting the project actually cleans up the telemetry state. Auto-detected based on skill install path.
- **Auto-detected `install_scope_path`.** Log script derives `user_home` vs `project_local` from the skill's install path — orchestrator doesn't have to pass it. Works across all three host conventions (`.claude`, `.codex`, `.gemini`).
- **Per-(run_id, event_type) dedup** for terminal events (`run_started`, `run_completed`, `run_failed`). If an orchestrator retries a phase after a transient error, the second fire is suppressed silently. `placeholder_fill_choice` is intentionally exempt (fires N times per run).
- **Model tracking.** Events now carry a `model` field (e.g. `claude-opus-4-7`, `gpt-5-codex`, `gemini-2.5-pro`) so the maintainer can answer questions like "which model produces the highest fit scores" or "which model hits this bug most". Self-reported by the orchestrator LLM on every event. Migration 008 adds the column; edge function propagates it with a 40-char cap; `--model` flag added to `bin/resumasher-telemetry-log`. PRIVACY.md updated to disclose.
- **Phase 9 underfill fixed.** `run_completed` now carries `used_multirole_format` alongside the existing `tailor_completed` event, so dashboards don't have to join events by `run_id` to see whether the multi-role rendering path was exercised.

### Data philosophy

- **Raw in, curated out.** The fit-analyst prompt asks the LLM for structured enums (`SENIORITY`, `RECOMMENDATION`); stronger models (Claude Opus, Gemini 2.5 Pro, GPT-5 Codex) comply reliably; weaker models (Haiku, `-mini`) paraphrase. Rather than null-out non-conforming values at ingest, the edge function now stores whatever the LLM emitted (lowercased + length-capped). Pipeline views downstream normalize via `CASE WHEN`. Event type is the only enum that stays validated because the schema shape depends on it.

### Fixed during live multi-host testing

- Phase 1 `mkdir -p .resumasher/run/` now happens BEFORE `jd.txt` is written (Gemini retry case).
- `$RUN_DIR` re-derived at the start of every phase's telemetry block — shell state doesn't persist across Bash tool calls.
- Empty `start-ts.txt` content now treated the same as a missing file (fall back to END_TS) so `duration_s` never ends up as a 56-year unix-epoch-sized value.
- `count_placeholders()` in Phase 9 no longer doubles stdout on zero matches (replaced `|| echo 0` with `|| true`); the log script's `f_num` helper defensively takes the first whitespace-delimited token of any numeric input as a second layer of protection.
- Consent prompt reorder: Off first (highlighted default), no "(Recommended)" label on Community — GDPR Article 7 says pre-selected yes + press-Enter is NOT valid consent.

### Security

- **RLS locked down to deny-all for the anon role** on both `telemetry_events` and `installations`. The edge functions use `SUPABASE_SERVICE_ROLE_KEY` internally to bypass RLS for validated writes; the anon key is purely a gateway ticket for `verify_jwt`. Students (and anyone with the public anon key) cannot read, insert, update, or delete directly via the REST API. Verified end-to-end: direct `POST /rest/v1/telemetry_events` with the anon key returns `42501 permission denied`.
- **Views recreated with `security_invoker=true`** so aggregate views respect the caller's RLS instead of running as their owner. Belt-and-suspenders over the existing `REVOKE SELECT FROM anon`.
- **`run_telemetry_retention()` has a pinned `search_path`** (`public, pg_catalog`) to close the Postgres "mutable search path" advisory-linter finding.

## [0.1.0] — 2026-04-18

First public release. Built in a single design-through-ship session; the CHANGELOG below bundles every meaningful change that landed between the initial commit and the first student-ready state.

### Added

- **Claude Code skill `/resumasher <job-source>`** with a nine-phase pipeline: first-run setup → intake → folder + GitHub mine → fit analysis → company research → tailor → parallel cover-letter + interview-prep → interactive placeholder fill → PDF render → log + summary. `<job-source>` accepts a file path, a URL, or literal pasted JD text.
- **EU + US resume styles** with ATS-safe single-column layout. Style choice persists in `.resumasher/config.json`. `--style` flag overrides per run.
- **Pure-Python PDF renderer** (`scripts/render_pdf.py`) using reportlab. No native deps — `pip install` just works on macOS, Linux, Windows. Bundled DejaVu Sans font handles non-ASCII names (Björn, François, Jiří) without box characters.
- **Multi-role tenure rendering.** When a candidate held multiple titles at one company (Meta → Senior Director → Director → Manager), the PDF shows ONE company entry with sub-role sub-bullets, preserving the promotion narrative.
- **Photo support for EU resumes.** Photos auto-downscaled to 500px max dimension, re-encoded as JPEG q=85 before embedding. Output PDFs stay under 200KB even from 3000×4000 phone-camera source photos.
- **`resume.pdf` accepted directly** when no markdown source exists. pdfminer.six extracts selectable text; image-only scanned PDFs fail with a clear error pointing at OCR or manual markdown as alternatives.
- **GitHub profile mining** during first-run setup. Prefers `gh api` (student's existing auth, 5000/hr limit), falls back to unauthenticated `urllib` (60/hr). Fetches non-fork, non-archived, non-empty repos sorted by most-recently-pushed; caps at 15 by default. Returns prose summaries with stars, topics, last-push date, and README content. 1-hour cache under `.resumasher/github-cache/<username>.json`. All failures non-fatal.
- **Interactive placeholder fill** before PDF render. When the tailor emits `[INSERT TEAM SIZE]` etc. for metrics the evidence didn't supply, Phase 7 prompts the student per-bullet with three options: paste specifics, soften to a no-metric version (pre-computed by the tailor as `<!--SOFT:...-->` alternates), or drop the bullet. No placeholder ever ships to the rendered PDF for resume or cover letter. Interview-prep placeholders stay as-is — those are practice prompts the student prepares before the interview.
- **Fit assessment** as a mandatory gate before tailoring. Honest 0–10 score with strengths and gaps. Doesn't block a low-fit application, but names the gap so the student applies with eyes open.
- **Company research** phase fetches 3–5 recent facts with inline citations, used by the cover-letter sub-agent.
- **`.resumasher/history.jsonl`** local application log. One line per application with timestamp, company, fit score, style, output dir, errors. Local-only, gitignored by default.
- **Golden fixtures** (`GOLDEN_FIXTURES/`) — sample MS Business Analytics portfolio with three projects (capstone, ML final, text mining), a non-ASCII student name (Ana Müller), and a realistic Deloitte Vienna JD. Students can try the skill end-to-end before pointing it at their own data.
- **`bin/resumasher-exec`** self-locating wrapper. Finds its own SKILL_ROOT and execs the venv Python with the right script path. Lets SKILL.md use `"$RS" orchestration <subcommand>` instead of verbose `"$PY" "$SKILL_ROOT/scripts/orchestration.py"` invocations.
- **Install script** (`install.sh`) — creates venv, installs requirements, chmods `bin/` wrappers. Idempotent. Actionable network-failure error message (as of this release) pointing students at retry options instead of a raw Python traceback.
- **`docs/DESIGN.md`** — design rationale extracted from the `/office-hours` and `/plan-eng-review` sessions that produced the skill. Frozen-ish; contributors read it before large PRs to understand why the skill is shaped the way it is.
- **Re-render workflow** documented in SKILL.md. Students who edit `tailored-resume.md` or `cover-letter.md` after a run ask Claude to "re-render the resume PDF" and Claude runs just the render step without re-dispatching sub-agents.

### Security

- **Sub-agent tool permissions tightened.** All six LLM sub-agents (folder-miner, fit-analyst, company-researcher, tailor, cover-letter, interview-coach) have explicit "do NOT use Bash, Read, WebFetch, Write, Edit, Grep, Glob" instructions in their prompts. Only company-researcher retains WebSearch + WebFetch (needed for its job). Defense-in-depth against prompt injection through `<<<UNTRUSTED_JD_*>>>` markers.
- **Prompt-injection markers** wrap JD content and company-research output before they reach any sub-agent. Markers are the primary defense; sub-agent tool restrictions (above) are the belt-and-suspenders layer.
- **`.claude/` directory ignored** by the folder miner. Previously, a project-scope install at `<project>/.claude/skills/resumasher/` had the folder miner walking its own source tree and presenting it to the fit-analyst as student evidence — a major data-contamination bug. Fixed via `DEFAULT_IGNORE_DIRS`.
- **`.gstack/` added to .gitignore** so security reports from `/cso` don't get accidentally committed to the public repo.
- **No secrets, PATs, or PII in the repo or its git history.** Verified via `git log -p --all` scan for known secret prefixes (ghp_, sk-, AKIA, etc.) at release time.

### Fixed

- **`ModuleNotFoundError: No module named 'scripts'`** when running `orchestration.py` directly from the CLI (not via `python -m`). Caused by `from scripts import github_mine` requiring the package path to be on `sys.path`. Fixed by inserting the script's own dir into `sys.path` at module load, and switching to sibling-style `import github_mine as gm`. Subprocess-level regression test added.
- **Shell state non-persistence between Bash tool calls.** Earlier SKILL.md assumed `SKILL_ROOT` and `PY` variables persisted across Bash tool invocations; they don't. Every invocation now starts with a path-resolution prologue that re-discovers SKILL_ROOT and errors clearly (pointing at `install.sh`) if the venv is missing.
- **`AskUserQuestion` with 1 real option crashed** with `InputValidationError: Too small: expected array to have >=2 items`. First-run-setup free-text fields (phone, location, photo path, GitHub) now use 2-option patterns with Other for the real input.
- **Markdown bold (`**text**`) rendered as literal asterisks** in PDFs. Renderer now converts `**x**` → `<b>x</b>` via reportlab's HTML subset before Paragraph parsing. Lone single asterisks (e.g., "v1.0*") are preserved, not false-matched as italic.
- **`/tmp/resumasher-*.txt` files polluted global `/tmp`** across runs and across users on shared machines, leaking student resume content. Intermediates moved to `$STUDENT_CWD/.resumasher/run/`, wiped at the start of each run, gitignored by default.
- **1MB resume PDFs** when a student's photo was a 3000×4000 phone-camera export. Photos downscaled to ≤500px, re-encoded as JPEG q=85. Output PDFs now ~150KB.
- **`install.sh` rejected project-scope installs** when run with a relative path arg due to relative-vs-absolute path comparison. Install.sh no longer accepts a target arg at all; it runs in place of wherever it was cloned.
- **First-run setup asked the same free-text value across TWO rounds** — one to say "I'll provide it," another to actually collect the value. Now one round: Skip / Use default / Other-field-for-the-real-value. Setup dropped from 3 rounds to 1-2 for most students.
- **Removed gstack-required notices and hooks** that were inherited from the development environment. Students don't need gstack to use resumasher.
