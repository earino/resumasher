## [Unreleased]

### Fixed

- **Sub-agent prompts no longer leak student PII into `/tmp`** ([#45](https://github.com/earino/resumasher/issues/45)). Real-run testing of #43 surfaced that some agents (Claude Code's Bash tool was the observed offender on macOS) improvised around SKILL.md's prompt-build pattern: instead of capturing the rendered prompt in a shell variable (which can hit argv-length caps for the folder-miner prompt at 100KB+), they wrote it to `/tmp/<kind>-prompt.txt` and passed the path. That leaked the student's resume + JD + project content as plaintext PII into a directory that's world-readable on macOS (mode 0755) until reboot. Belt-and-suspenders fix: **(suspenders)** SKILL.md's "Sub-agent prompt pattern" section now explicitly prescribes `$RUN_DIR/prompts/<kind>.txt` for prompt staging â€” gitignored, run-scoped, wiped each run â€” and explicitly forbids `/tmp/` for this purpose with the privacy reasoning called out so future maintainers don't drop it. **(belt)** New `cleanup_stray_prompts()` function + `cleanup-stray-prompts` CLI subcommand modeled on `cleanup_stray_outputs` from #29. Scans `/tmp` (top-level only, never recursive) for files whose basenames match `<kind>-prompt.{txt,md}` or `<kind>_prompt.{txt,md}` for a registered prompt kind AND whose mtime is newer than `$START_TS`, then deletes them unconditionally (no recovery value â€” they're transient sub-agent intermediates containing PII we want gone). Wired into SKILL.md Phase 8 alongside the existing `cleanup-stray-outputs` invocation. Pattern matcher accepts both hyphen-separator (`folder-miner-prompt.txt`) and underscore-separator (`folder_miner_prompt.txt`) stem forms â€” both shapes appeared across different real-run sessions. 16 new deterministic tests covering each registered kind across both separators, mtime gate, suffix gate, top-level-only scope, generic /tmp file immunity (bash_history, screenshots, near-miss names), missing scan dir, CLI JSON summary, and SKILL.md prescription assertions (the prescribed `$RUN_DIR/prompts/` line + explicit forbid of `/tmp/` + the cleanup invocation must all stay in SKILL.md across future edits).

### Changed

- **Stacked-date layout for resume PDF block titles** ([#42](https://github.com/earino/resumasher/issues/42), closes [#22](https://github.com/earino/resumasher/issues/22), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max) â€” #22 surfaced six concrete typography complaints, four shipped in v0.4.0 via #28; this is the last one). Block and sub-block titles that contain a recognizable date segment now render as two paragraphs â€” title above, date below in a slightly smaller (9.5pt) regular-weight metadata style â€” instead of inlining the date in parentheses. Targeted fix for the last remaining piece of #22 (right-aligned dates), reframed for ATS safety: single-flow text only, no Tables or tab stops, because [Jobscan testing](https://www.jobscan.co/blog/resume-tables-columns-ats/) finds 2-column / Table layouts are the most common cause of word-salad parsing in commercial ATS systems (Workday, Greenhouse, Taleo, iCIMS). Resumasher's north-star metric is "the application gets through the parser," not "the PDF could win a typography award" â€” so we trade the literal right-aligned ask for a stacked-line variant that preserves the typographic gesture (real size hierarchy between title and metadata) while reading linearly to every parser. Detection heuristic: parenthesized-at-end or pipe-separated segment containing a 4-digit year (1900-2099). Bare years in prose ("2024 Economic Survey â€” Statistics Austria") are NOT matched, so publication titles aren't hijacked. Titles without a recognizable date fall through to single-paragraph rendering â€” zero behavior change in that case. New `BlockMetadata` and `SubBlockMetadata` paragraph styles (9.5pt regular, body-black, sub-block indent matches `SubBlockTitle`). **Bundled correctness fixes:** the stacked-date pattern surfaced two pre-existing layout problems that would have shipped a regression on multi-role resumes â€” bullets under sub-blocks rendered with the bullet character to the LEFT of the sub-block content above (Bullet.bulletIndent=4 vs SubBlockTitle.leftIndent=8, glaring once a metadata line sat directly above the bullet), and SectionHeading.spaceBefore=10 felt proportionally compressed once each block gained a metadata line. Distinct from the in-content `---` markdown rule (`thickness 0.5pt`, color `#888888`) which is a softer in-section break â€” section dividers are slightly thicker and darker to read as structural. ATS round-trip preserved: `HRFlowable` produces no text in the PDF text layer, so parser extraction is unaffected. Also applied to interview-prep document rendering where it uses `SectionHeading`. **Polish pass:** `BlockMetadata` and `SubBlockMetadata` colored `#555555` so date/location lines recede as metadata rather than competing with bullets for attention; `Contact` and `SubheadCenter` (US-style centered contact) reduced to 9.5pt and recolored `#555555` so the contact line reads as header rather than body; `BlockTitle.spaceBefore` bumped 4 â†’ 6 for natural breathing room between consecutive jobs within a section. **Clickable links anywhere a URL appears:** `_linkify_contact()` was generalized to `_linkify_text()` and applied to every Paragraph render path â€” block titles (so a project entry like `### Resumasher (github.com/earino/resumasher)` produces a clickable link), sub-block titles, bullets (so a bullet like "see https://example.com/docs" is clickable), and Summary-style body paragraphs. Pattern set unchanged: email, `linkedin.com/in/X`, `github.com/X`, explicit `https://...` URLs. Bare-domain forms get auto-prefixed with `https://` on the href while displayed text stays bare-domain. Phone, location, prose, and any non-URL text pass through unchanged. **Page-break correctness via per-sub-block `KeepTogether`:** real-run testing surfaced that a multi-role block (e.g., one company with 3 sub-roles, each with its own bullets) was wrapped in a single giant `KeepTogether` flowable. When the entire block exceeded the remaining page space, reportlab page-broke BEFORE the whole thing â€” leaving the bottom of page 1 blank. Fixed by emitting per-sub-block `KeepTogether` groups: block title (+ direct bullets, if any) gets one group, each sub-block (+ its bullets) gets its own. Page breaks now occur between sub-blocks â€” the natural typesetting boundary â€” while sub-role titles stay glued to their own bullets. Single-role blocks (no sub-blocks) still emit exactly one `KeepTogether` each â€” no over-fragmentation. **Final visual polish:** (1) markdown code-span backticks (the tailor LLM commonly wraps URLs in `` ` ``-pairs like `` `github.com/foo` ``) are now stripped at render time so the rendered output has no decorative backticks; reportlab Paragraph doesn't render code spans natively and there's no use for the markers in a resume. (2) Link `<a>` tags now carry an explicit `color="#0645AD"` (muted Wikipedia-blue) so URLs are visually discoverable to recruiters skimming the PDF rather than blending with body text â€” earlier "minimalist, links inherit color" call was reversed after real-run review showed clickable URLs went unnoticed. (3) URL regex character class tightened to exclude every character RFC 3986 says is never in a URL (whitespace, `<>"`{}^|\`) plus closing paren / bracket â€” backtick exclusion specifically fixed a bug where href values gained a trailing `` ` `` baked into the link target. **Project titles shaped `Name (URL)` collapse to a single clickable Name.** Pre-fix, a project entry like `### ci_fixer_bot (github.com/earino/ci_fixer_bot)` rendered as `ci_fixer_bot (github.com/earino/ci_fixer_bot)` with only the URL portion in blue â€” explicit URL display next to the project name was visual noise. Now the entire title becomes a single clickable Name in muted blue (the URL still lives in the `<a>` href attribute, ATS extraction still finds the link). New `_linkify_title()` (used only in titles, not bullets â€” bullets routinely have parens with extra metadata like star counts and we want to keep the inline-URL render there). Title-collapse triggers only when parens content is EXACTLY one of our URL patterns; titles like `Project (Aug 2022 â€“ Aug 2025)` (date in parens, handled by date split) and `Project (github.com/me/x, 23 stars)` (URL plus extra prose) are unaffected. **Date split now works for titles with trailing text after the parens.** Pre-fix, the parens-with-date regex was anchored to end-of-string (`\s*$`), so a title shaped `Senior Director, Data Science (August 2022 â€“ August 2025) â€” Zurich` (closing paren NOT at end â€” `â€” Zurich` trails it) silently failed to split: the date stayed inlined and only titles with parens at the very end (CEU-style: `Visiting Professor (December 2017 â€“ July 2025)`) got the stacked-date treatment. The end-anchor was wrong from the start; real-run resumes routinely have a city/location between sub-blocks and the parent block. Removed the anchor and changed the title rebuild to preserve text on BOTH sides of the parens â€” title becomes `Senior Director, Data Science â€” Zurich` and the metadata line carries `August 2022 â€“ August 2025`. Five regression tests pin the shape including parens-at-start, parens-at-middle, parens-at-end, and an end-to-end render of the exact broken case from Eduardo's run. **Italic markdown finally renders as italic.** Pre-#42 the renderer skipped `*italic*` conversion for fear of false-matching single asterisks in prose ("footnote*"), so course titles like `*Applied Deep Learning*` (common in tailored bullets) shipped with the asterisks visible as decorative characters. Earlier "save 1.4MB on repo size" call (PR #30, deferred to a future LaTeX track) was the wrong tradeoff â€” visible asterisks in shipped resumes vs. negligible repo growth alongside a 50MB venv. Link styling: no `color=` attribute on the `<a>` tag, so links inherit the surrounding paragraph color (the soft gray of the contact line) â€” no 1990s blue underline, modern resume convention. Verified end-to-end: PDF Link annotations actually present in the output PDF (pdfminer pulls `mailto:`, `https://linkedin.com/...`, `https://github.com/...` from the rendered file). ATS round-trip preserved â€” link annotations are a metadata layer; the underlying text stream is identical to the pre-link version. 14 new deterministic tests including pattern recognition for parens/pipe/year-range/Present forms, no-date regression guard, sub-block treatment with correct indent, ATS round-trip assertion that all date strings remain extractable in document order, sub-block bullet indent invariant, top-level bullet regression guard, and section-separation lower bound. Closed [#31](https://github.com/earino/resumasher/issues/31) (LaTeX renderer track) as not-planned in the same arc â€” engine swap was disproportionate to the actual problem.

### Changed

- **`install.sh --dev` flag for contributors.** Bare `install.sh` still installs runtime deps only (students don't need pytest/jupyter â€” keeps their first install fast and lean). Contributors running the test suite now install with `bash install.sh --dev`, which pulls in `requirements-dev.txt` (runtime + pytest + jupyter). ### Fixed

- **interview-coach sub-agent no longer pollutes student CWD on weaker models** ([#29](https://github.com/earino/resumasher/issues/29)). On Haiku 4.5 (does not reproduce on Opus 4.7), the interview-coach sub-agent ignored the buried `TOOL USAGE CONSTRAINTS` block and used the Write tool to plant a markdown file with a fabricated name (e.g. `Ana_Muller_Interview_Prep_Bundle.md`) directly in `$STUDENT_CWD` â€” a hard contract violation, since resumasher's documented output location is `applications/<slug>-<date>/`. Defense-in-depth fix across four surfaces: **(1) Prompt surgery** in `scripts/prompts.py::INTERVIEW_COACH_PROMPT` â€” opening changed from "Build an interview preparation bundle" to "Generate interview questions and answers", added explicit "Return the document as text in your response. Do not create any files on disk." to the first paragraph, moved the `TOOL USAGE CONSTRAINTS` block from the bottom (line ~550 of a 10KB prompt, AFTER the "produce a markdown document" framing had already shaped the model's plan) to immediately after the first paragraph, and strengthened the constraints to specifically name the Write tool. **(2) SKILL.md Phase 6 hardening** â€” replaced ambiguous "Save the outputs to `$OUT_DIR/cover-letter.md` and `$OUT_DIR/interview-prep.md`" with explicit instructions: take the sub-agent's text response and use the Write tool to save it; do NOT scan the filesystem for sub-agent-written files (that behavior IS the bug, not the recovery); reference issue #29 so future edits don't revert the explicit wording. **(3) Post-phase cleanup scan** â€” new `orchestration cleanup-stray-outputs` subcommand and `cleanup_stray_outputs()` Python function. Scans `$STUDENT_CWD` top level (never recursive) for `.md` files whose names contain "interview", "prep", or "bundle" (case-insensitive substring match) AND whose mtime is newer than the dispatch timestamp (passed via `--since-timestamp`). Pre-existing student files are immune via the mtime gate; documented input filenames (`resume.md`, `cv.md`, `jd.md`, etc.) are immune via a protected-names allowlist. If `$OUT_DIR/interview-prep.md` is missing/empty, MOVE the rogue file there (best-effort content recovery); otherwise DELETE it (orchestrator already wrote the right thing; rogue is pollution). Wired into SKILL.md Phase 6 to run after every interview-coach dispatch. **(4) Test coverage** â€” three new test files: `test_cleanup_stray_outputs.py` (12 deterministic tests covering DELETE/MOVE paths, mtime gate, name-pattern matching, protected-names guard, top-level-only scope, defensive missing-input handling, and CLI subcommand integration), `test_interview_coach_prompt.py` (6 structural assertions on the prompt text â€” first-paragraph framing, no-files imperative, constraints-before-emit ordering, Write-tool-named-explicitly, substitution-slot preservation, FAILURE-sentinel preservation), `test_skill_md_phase6.py` (5 wording assertions on SKILL.md). Plus a Tier 3 live LLM test (`test_interview_coach_live.py`) that invokes `claude -p --model claude-haiku-4-5` against the real interview-coach prompt with a synthetic-persona fixture set (Ana MÃ¼ller, MS Business Analytics at CEU Vienna â€” fully fabricated, safe to commit) and asserts zero Write tool calls + zero stray markdown files. The live test auto-skips when `claude` is absent (CI runners) or `RESUMASHER_SKIP_LIVE=1` (manual escape hatch); registered via the `live_llm` pytest marker in `conftest.py`. Runs automatically on dev boxes that have Claude Code installed; uses the developer's existing subscription so per-run cost is zero. The cleanup scan is the belt (catches rogue files even if a future model regresses against the prompt); the prompt surgery is the suspenders (stops Haiku from creating the rogue file in the first place); the live test is the one-time signal that the suspenders work. Per-run human verification on Haiku before merge is the acceptance criterion the issue asks for.
- **`discover_resume` preserves on-disk filename case** ([#27](https://github.com/earino/resumasher/issues/27)). On case-insensitive filesystems (macOS APFS default, Windows NTFS default â€” so ~100% of the cohort), `scripts/orchestration.py::discover_resume` iterated `RESUME_CANDIDATES` and returned `cwd / "cv.pdf"` when the on-disk file was actually `CV.pdf`: the probing `exists()` check matched case-insensitively but the returned `Path.name` reflected the candidate string we constructed, not the real filename. Functionally it still resolved to the right file (reads worked), but logs and telemetry showed the wrong case. Fix: enumerate the directory with `iterdir()` once, match on lowercased names against a priority map built from `RESUME_CANDIDATES`, and return the highest-priority match with its real on-disk `.name` intact. Priority order (markdown > PDF, resume > CV) is preserved. New regression test `test_discover_resume_preserves_on_disk_case_mixed` uses a name (`Cv.pdf`) that isn't literally in `RESUME_CANDIDATES`, so it catches the bug on case-sensitive filesystems (Linux CI) too â€” the old probing loop would have returned None. The existing `test_discover_resume_accepts_cv_pdf_variants` tightened from a case-insensitive `.lower() == "cv.pdf"` check to a strict `== "CV.pdf"`, which fails on the old code under Windows NTFS â€” the Windows CI matrix merged in #35 is now the regression gate where it matters most.
- **Windows test harness â€” telemetry tests invoke scripts via explicit `bash`.** Surfaced by the Windows CI matrix ([#34](https://github.com/earino/resumasher/issues/34)): `tests/test_telemetry_scripts.py::_run` invoked `bin/resumasher-telemetry-log` (a bash script) directly via `subprocess.run([str(bin_path), ...])`. On POSIX the kernel honors the `#!/usr/bin/env bash` shebang, but Python's `CreateProcessW` on Windows can only launch PE/EXE files â€” the script invocation failed with `OSError: [WinError 193] %1 is not a valid Win32 application` and ~25 tests errored out. Git Bash has its own shebang interpreter, but `subprocess.run` doesn't go through Git Bash. Fix: prepend `"bash"` to the argv so Python launches `bash.exe` (Windows) or `/bin/bash` (POSIX) and bash itself handles the script. Matches how students actually invoke the scripts. POSIX test behavior is unchanged. Production path was unaffected â€” `bin/resumasher-exec telemetry` dispatches through bash-to-bash, no Python subprocess involved.
- **Windows test harness â€” install-probe regression test skipped on Windows.** `tests/test_install_python_probe.py::test_install_sh_falls_through_when_python3_is_broken_ms_store_stub` simulates a Windows-only bug (Microsoft Store `python3` stub) on non-Windows by dropping fake bash-script shims on PATH. Python's `subprocess.run` on Windows can't execute bash scripts (same WinError 193 as above), so the test has no way to run its own simulation natively. Marked `@pytest.mark.skipif(sys.platform == "win32")`. Windows students still catch regressions against real MS Store stubs organically; this test provides Linux/macOS-side regression coverage.
- **Windows stdout/stderr encoding â€” CLIs no longer crash on `â†’`, `â€¦`, or non-ASCII names.** Surfaced by the new Windows CI matrix ([#34](https://github.com/earino/resumasher/issues/34)): `scripts/orchestration.py build-prompt` (and the other CLI entry points) wrote prompt templates containing `â†’` directly to stdout, but Python on Windows defaults stdout/stderr to the system ANSI code page (typically CP1252) when not attached to a TTY â€” which is exactly the shape every `resumasher-exec` invocation takes. Writing `â†’` raised `UnicodeEncodeError: 'charmap' codec can't encode character 'â†’'` and the phase died. Any Windows student running orchestration whose output included a `â†’`, `â€¦`, curly quotes, or a non-ASCII name hit this â€” the v0.4.1 install fixes got students past install but not past the first CLI call that produced these glyphs. Fix: call `sys.stdout.reconfigure(encoding="utf-8")` (and the same for stderr) at every `if __name__ == "__main__":` block in `scripts/` â€” orchestration.py, render_pdf.py, github_mine.py. New `tests/test_stdout_encoding.py` reproduces the Windows behavior on any OS via `PYTHONIOENCODING=cp1252`, pinning both the crash-free invariant AND the byte-level assertion that the arrow glyph makes it through as UTF-8 (not lossily transcoded to `?`).

## [0.4.1] â€” 2026-04-24

v0.4.1 is a Windows hotfix release. The cohort is ~90% Windows, and [@b0glarka](https://github.com/b0glarka) reported [#32](https://github.com/earino/resumasher/issues/32) â€” a fresh Windows 11 + Git Bash install of v0.4.0 failed before the skill could run. Three cascading Windows-specific bugs in `install.sh` were caught across two rounds of verification on her machine; all three are fixed here. Windows students on a fresh clone can now run `bash install.sh` end-to-end.

### Fixed

- **Windows (Git Bash) install and runtime** ([#32](https://github.com/earino/resumasher/issues/32), reported by [@b0glarka](https://github.com/b0glarka) with a three-file patch that pinpointed every failure site, then caught two additional Windows-specific failure modes during verification of the patch itself). Three cascading Windows bugs fixed together. **(1) Venv layout and interpreter name:** `install.sh` hardcoded `python3` and `.venv/bin/pip`; `bin/resumasher-exec` hardcoded `.venv/bin/python`; both SKILL.md prologues gated on `.venv/bin/python`. On Windows/Git Bash, `python3` is typically absent and CPython's venv layout uses `Scripts/` instead of `bin/`. Installer now prefers `python3` and falls back to `python`; `VENV_BIN` is derived as `bin/` or `Scripts/` after venv creation; wrapper and both SKILL.md prologues OR-join the POSIX and Windows venv-Python checks. **(2) Microsoft Store App Execution Alias stub:** Windows also ships a `python3` stub at `%LOCALAPPDATA%\Microsoft\WindowsApps\python3` â€” a real file on PATH that exits non-zero with "Python was not found" when invoked. A pure `command -v` check accepts the stub and aborts at `python3 -m venv`. Installer now probes each candidate by running a 3.10+ version check, disqualifying the stub and falling through to `python`. **(3) pip self-upgrade via the pip binary:** On Windows, `pip.exe` can't overwrite itself while running (file is locked), and pip detects this and refuses `pip install --upgrade pip` when invoked via the binary â€” redirecting to `python -m pip install --upgrade pip`, which copies pip to a temp path before replacing. Installer now invokes pip via `python -m pip install` uniformly, sidestepping the self-upgrade wall. `tests/test_install_python_probe.py` simulates the MS Store stub on any OS and pins both the probe-by-invocation and `python -m pip` invariants. No Windows CI matrix yet â€” the reporter smoke-tested the patch on her Windows 11 + Git Bash setup and surfaced both the MS Store stub bug and the pip-self-upgrade bug before merge.
- **SKILL.md prologue drift guard.** The two path-prologue blocks (top-of-file and the re-render section at L1026) had already silently drifted before #32 â€” Copy B was missing the "venv missing â†’ run install.sh" error branch that Copy A had. Any cross-cutting fix (like the Windows one above) has to touch both, and "forgot the second copy" is a silent-break class of bug. New `tests/test_skill_md_prologue.py` asserts exactly two prologues exist, both check POSIX and Windows venv-Python paths, and both scan the same candidate install roots. Future cross-cutting changes that touch only one copy will fail in CI instead of shipping.

## [0.4.0] â€” 2026-04-20

v0.4 was shaped almost entirely by CEU cohort feedback. Six cohort-reported issues shipped: photo aspect / placement / horizontal rules ([#22](https://github.com/earino/resumasher/issues/22)), contact-header enforcement ([#18](https://github.com/earino/resumasher/issues/18)), bullet-to-title attachment ([#19](https://github.com/earino/resumasher/issues/19)), photo path persistence ([#20](https://github.com/earino/resumasher/issues/20)), JD preservation ([#15](https://github.com/earino/resumasher/issues/15)), and local-venv test resolution ([#24](https://github.com/earino/resumasher/issues/24)). Plus agent-driven bug investigation: students describe a problem in plain English, the agent investigates via `orchestration inspect`, matches against [KNOWN_FAILURE_MODES.md](docs/KNOWN_FAILURE_MODES.md), and files a structured bug report â€” no `--debug` flag to remember. The typography taste-call half of #22 (font hierarchy, right-aligned dates) was attempted and backed out; tracked in [#31](https://github.com/earino/resumasher/issues/31) as an optional LaTeX-renderer track for a future release.

### Added

- **Agent-driven bug investigation.** Students in an AI CLI can now report issues in plain English ("my photo looks stretched," "my name is missing from the PDF") and the agent follows a new [debugging playbook](SKILL.md#debugging-this-skill) to investigate its own output. New `orchestration inspect --resume/--pdf/--photo` subcommands return structured JSON (parse tree, section order, photo dimensions) with lightweight warnings for the most common bug signatures. A [`docs/KNOWN_FAILURE_MODES.md`](docs/KNOWN_FAILURE_MODES.md) catalog lists catalogued bugs with symptom + signature + fix location; agents consult it as a checklist during diagnosis. [`docs/BUG_REPORT_TEMPLATE.md`](docs/BUG_REPORT_TEMPLATE.md) provides the skeleton the agent fills in and submits via `gh issue create` after getting student consent. Contact info (name, email, phone, LinkedIn, GitHub) is redacted by default; company names and metrics stay for reproduction value. No `--debug` flag: the agent is the interface. Inspired by feedback from [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max), a CEU cohort student whose bug report made it obvious students needed an easier way to help us debug their issues. Thanks for the push.
- **Persist JD to application folder as `jd.md`** ([#15](https://github.com/earino/resumasher/issues/15), reported by [@b0glarka](https://github.com/b0glarka) â€” first v0.4 feature shaped by actual cohort feedback). The job description now lands in `applications/<slug>-<date>/jd.md` alongside the other artifacts, so students can re-read the exact wording of a requirement, open it next to `interview-prep.md`, or grab the source URL for a recruiter follow-up â€” weeks after the run, and even after the `.resumasher/run/` scratch directory has been wiped by subsequent runs. Done at Phase 3 (right after `mkdir -p "$OUT_DIR"`) rather than Phase 9, so the JD survives even if a later phase (tailor, PDF render) hard-stops. For URL-mode inputs, a `Source URL: <url>` header is prepended by a new `format-jd` orchestration subcommand at Phase 1, so the posting URL is preserved into `jd.md` even if the posting page is later taken down. File-mode and literal-mode inputs carry no URL (nothing to preserve).
- **End-of-run tweak tip for external editors.** Phase 9 Next steps block now includes a friendly pointer for students who want to adjust fonts, spacing, or colors: open the PDF in Google Docs, Pages, Word, or Canva. The resumasher PDF ships ATS-safe and content-correct; cosmetic polish is a natural handoff to tools students already know.

### Fixed

- **Photo area: aspect preserved, placement configurable, horizontal rules rendered** ([#22](https://github.com/earino/resumasher/issues/22), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Three visible PDF fixes shipped together. - **Photo path persisted to tailored markdown** ([#20](https://github.com/earino/resumasher/issues/20), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Previously, a student who provided a photo got it embedded in `resume.pdf` but the source `tailored-resume.md` had no record of the photo path â€” re-render after manual markdown edits required config or a CLI flag, making the markdown non-self-describing. - **Local test suite now works in a macOS venv** ([#24](https://github.com/earino/resumasher/issues/24)). Tests that shell out via `subprocess.run(["python", "-m", ...])` were spawning a subprocess that resolved `python` through `$PATH`, which typically finds system Python (not the venv's). The venv's installed packages (reportlab, pdfminer.six, etc.) weren't visible to the subprocess, and ~15 tests failed with `ModuleNotFoundError`. No behavior change in CI; local test runs now just work.
- **Bullets stay attached to their project titles** ([#19](https://github.com/earino/resumasher/issues/19), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Previously, when the tailor emitted `**Project Name** | metadata` directly under `## Research Experience` (or Work Experience, or similar) without an intervening `### Company` heading, `parse_resume_markdown` silently dropped the titles into `raw_paragraphs` and bullets into `raw_bullets`. The renderer stacked all project titles at the top of the section then dumped all bullets at the bottom in a single flat list â€” readers could not tell which bullet belonged to which project. The parser now synthesizes a `ResumeBlock` when it sees `**Title**` with trailing metadata directly under `##`, and bullets attach to that synthetic block until the next `**Title**` or `##`. The trailing-metadata check guards against hijacking bold-only prose in Summary sections (e.g., `**Accomplished data leader.**` still renders as a paragraph, not a block title). Existing `### Company` + `**Role**` sub-block behavior for multi-role tenures is unchanged. Six new parser + renderer tests guard against regression, plus the previously-fired `ORPHANED_BULLETS` inspect warning is now proven to stop firing for this shape.
- **Rendered PDFs can no longer ship without a candidate name** ([#18](https://github.com/earino/resumasher/issues/18), reported by [@guiqvlaixi2164-max](https://github.com/guiqvlaixi2164-max)). Previously, when the tailor emitted a pipe-separated contact line on line 1 without a `# Name` H1 prefix, `parse_resume_markdown` silently dropped the line, producing a PDF with no name, email, phone, or LinkedIn anywhere â€” an ATS cannot identify the applicant, the application is silently filtered out, the student never hears back. The renderer now raises `MissingContactHeaderError` before writing any PDF, with a student-facing error that names the expected shape, shows the first line the parser actually saw, and points at KNOWN_FAILURE_MODES.md #1 for the tracking issue. Tailor prompt hardened to explicitly forbid any other line-1 shape. Exit code 2 on failure so orchestration knows to surface the error. Five new regression tests in `tests/test_render_pdf.py` guard against reintroducing the silent-drop behavior.

### Data philosophy

- **Raw in, curated out.** The fit-analyst prompt asks the LLM for structured enums (`SENIORITY`, `RECOMMENDATION`); stronger models (Claude Opus, Gemini 2.5 Pro, GPT-5 Codex) comply reliably; weaker models (Haiku, `-mini`) paraphrase. Rather than null-out non-conforming values at ingest, the edge function now stores whatever the LLM emitted (lowercased + length-capped). Pipeline views downstream normalize via `CASE WHEN`. Event type is the only enum that stays validated because the schema shape depends on it.

### Security

- **RLS locked down to deny-all for the anon role** on both `telemetry_events` and `installations`.
